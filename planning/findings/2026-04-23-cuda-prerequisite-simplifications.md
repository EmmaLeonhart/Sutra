# CUDA-prerequisite simplifications (2026-04-23)

Mechanical cleanup pass on the numpy backend to reduce the surface
area of the PyTorch/GPU port (STATUS queue item 2). Five items,
all landed as separate commits on `claude/cuda-simplification-review-P2lkt`.

## Why this happened now

The PyTorch/GPU port is the next piece of work in STATUS item 2, and
prior "get GPU ready" work (fused bundle_of_binds, vectorized
argmax_cosine, disk-cache plumbing) already landed. The blocker was
no longer correctness — the compiler emits the right shapes — but
structural: *writing* the new backend would mean re-doing a lot of
work that should have been written once. This pass attacks that
structural debt.

Shape of the diff:

- `codegen_numpy.py`: 652 → 185 lines (-72%)
- New file `runtime_numpy.py`: 404 lines of real, lint-able,
  testable Python
- `codegen_flybrain.py`: +41 lines net (extracted CodegenBase in
  place, added sibling FlyBrainCodegen subclass)
- Test corpus: 148 passed / 50 subtests, unchanged throughout

## Items

### 3a. Extract `_NumpyVSA` runtime into `runtime_numpy.py`

The entire VSA runtime (`_NumpyVSA` class plus the `_argmax_cosine`
/ `_select_softmax` / `_vector_map_lookup` helpers) used to live
inside `codegen_numpy.py` as ~500 lines of `self._emit("...")`
calls. The file could not be linted, type-checked, or tested as
Python — only as a string-building pass. A PyTorch backend would
have required copy-pasting the whole emit stream into a parallel
`codegen_torch.py`.

Moved the runtime to a real source file. `codegen_numpy._emit_prelude`
reads `runtime_numpy.py` at emit time, strips its own module
docstring (replaced by the "Generated by ..." docstring), and inlines
the remainder verbatim.

Emitted output diff vs. baseline: the `_VSA = _NumpyVSA(...)` and
`_VSA.embed_batch([...])` lines move from between the class and the
helpers to after the helpers. Semantics identical (helpers reference
only `_np`; `_VSA` is only used in user code that follows).

### 3b. Deduplicate runtime helpers

Two pairs of near-identical code folded onto shared helpers:

- `embed()` and `embed_batch()` both mean-center / normalize /
  pad-or-truncate / renormalize raw Ollama vectors. Extract
  `_postprocess_embedding(v)`. Keeps single vs. batched fetches
  byte-identical by construction.

- `_argmax_cosine(query, candidates)` and `_vector_map_lookup`'s
  cosine fallback both stack rows / compute norms / guarded cosine
  / argmax. Extract `_argmax_cosine_idx(M, q) -> int | None`; each
  caller handles the "which value" mapping and the None-on-zero-
  query fallback locally.

One test assertion that had been looking for `keys = ...` / `scores
= (keys @ q) / ...` in `_vector_map_lookup` was rewritten to assert
on the shared helper's shape (`M`-parameter + `def
_argmax_cosine_idx`).

### 3c. Vectorize `_NumpyVSA.loop` prototype match

The old implementation called `self.similarity(state, proto)` inside
a Python for-loop over `compiled_prototypes` — P dot products, P
norm computations, P small kernel launches per iteration.

Hoisted the prototype stack out of the loop: stack into `(P, d)`
once at entry, precompute `proto_norms` once. Per iteration: one
matmul `P @ state` + argmax. `state` is already unit-norm after its
own renormalize, so no division by its norm. Zero-norm prototypes
still guarded to `-inf`.

Same shape the PyTorch backend will reuse unchanged. Preserved
semantics on edge cases: empty `compiled_prototypes` falls through
with `best_name=None` (previously would `NameError` on the tail
return if `max_iters=0`); non-empty with no match returns the last
iteration's best as before.

### 3d. `make_random_rotation` — `inv(V)` → `V.conj().T`

**Self-caught shortcut.** Initial STATUS writeup for this item
claimed the eigendecomposition fractional-power branch was dead
code (no demo uses a non-π angle). That audit was wrong: all the
loop-using demos (`concept_search.su`, `counter_loop.su`,
`loop_rotation.su`) compile to `make_random_rotation(angle=1.0,
...)` — which IS a non-π angle, so the eigendecomposition branch
runs on every loop construct. Dropping it would have broken live
code. Noted in the 3d commit message and in the STATUS.md item per
CLAUDE.md's "self-caught shortcut" rule.

Real simplification that does apply: `Q` is real orthogonal (hence
normal), so its eigenvector matrix `V` from `_np.linalg.eig(Q)` is
unitary, and `_np.linalg.inv(V) == V.conj().T` to machine precision.
Verified: `||inv(V) - V.conj().T||_inf ~ 8e-15`. Swapped the O(d³)
explicit inversion for the O(d²) conjugate transpose. Output agrees
to 1e-15.

Not a CUDA-critical path — `make_random_rotation` runs once per
loop construct at R-construction time (host-side at program start),
not per iteration. The eigendecomp cost doesn't hit GPU hot path.
Pure quality-of-implementation.

The non-orthogonality of R at fractional angles (`|R R^T - I| ~
0.08`) is inherent to the fractional-power construction for real Q
with unpaired real eigenvalues (e.g. a ±1 eigenvalue raised to a
non-integer power becomes complex; `_np.real(R)` then drops the
imaginary component and orthogonality is lost). This was present
before the change and is not fixed by it; flagged for whoever
eventually replaces fractional-Q with a skew-symmetric-exponential
construction.

### 3e. Flatten `NumpyCodegen → FlyBrainCodegen` inheritance

Before: `NumpyCodegen(FlyBrainCodegen)`. The numpy backend inherited
from fly-brain purely to reuse AST-walking code; the two emit
fundamentally different runtimes. The inheritance bled fly-brain
state into numpy init — `NumpyCodegen.__init__` had to pass
`runtime_n_kc=0` and `runtime_use_hemibrain=False` to
`super().__init__` — and a future `TorchCodegen` would have had to
pick between Numpy and FlyBrain as parent, inheriting whichever's
baggage.

After: extracted `CodegenBase` in `codegen_flybrain.py` with all the
substrate-neutral scaffolding (AST walkers, builtin dispatch,
bounded/while/for loop shapes, `_next_loop_id`, `_extract_*`).
`FlyBrainCodegen(CodegenBase)` and `NumpyCodegen(CodegenBase)` are
siblings. `CodegenBase._emit_prelude` and
`CodegenBase._translate_eigenrotation_loop` raise
`NotImplementedError` so accidental instantiation of the base fails
loudly rather than emitting junk. `_loop_counter` moved to
`CodegenBase` so subclasses share a unique-ID namespace.

Verified: `NumpyCodegen` instances no longer carry `runtime_n_kc`
or `runtime_use_hemibrain`. `NumpyCodegen.__mro__` is
`(NumpyCodegen, CodegenBase, object)` — no longer through
`FlyBrainCodegen`.

## Collective effect on the PyTorch port

Before this pass, writing `codegen_torch.py` would have meant:

1. Decide whether to inherit from `NumpyCodegen` (gets the
   `_UNSUPPORTED_BUILTINS` override but also inherits fly-brain
   state) or from `FlyBrainCodegen` (gets fly-brain state directly).
2. Copy-paste the ~500-line emit stream from `codegen_numpy.py`,
   swapping `_np` → `_torch` across every string literal.
3. Hope no typos.

After this pass:

1. Subclass `CodegenBase` directly.
2. Write `runtime_torch.py` as real Python (lintable, testable).
3. Point `_RUNTIME_PATH` at it.
4. Override `_translate_eigenrotation_loop` for
   substrate-appropriate rotation parameters (or reuse numpy's by
   calling into it).

The fused shapes (`bundle_of_binds`, `_argmax_cosine_idx`, the
vectorized loop match) already match what CUDA wants — those were
prior work (STATUS item 2's "Aggressive simplification + fused
codegen" commit). This pass makes the *scaffolding* reusable too.

## What's left blocking the port

Still STATUS item 2's footnote: generalized ANF + dep analysis is
not done. Only the `bundle(bind,bind,...)` pattern fuses; other
potentially-independent sequences (e.g. `bundle(bind(r,f), c,
bind(r2,f2))`) still emit sequentially. For the three demo
programs the fused shapes cover the hot path; larger programs may
hit the sequential fallback and want broader dep analysis before
the GPU port shows end-to-end speedups.
