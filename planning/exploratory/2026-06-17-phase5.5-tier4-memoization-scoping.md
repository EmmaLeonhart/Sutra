# Phase 5.5 tier 4 — dynamic multiple recursion → automatic memoization: scoping

**Date:** 2026-06-17
**Status:** scoping / design. The last tier of the recursion-execution model
(`planning/sutra-spec/recursion-execution-model.md`). Tiers 1–3 done (tail/loops → recurrent
neurons; single non-tail → tail; fixed-depth multiple → opt-in compile-time pre-eval). Tier 4 is the
hardest and least-specified — the spec gives the vision ("memo table = a lazy lookup table / DAG,
realized as recurrent-neuron state, naive fib → linear, stays native") but not the concrete
substrate mechanism. This scopes it before coding.

## The target

Dynamic multiple recursion — multiple recursive calls whose controlling argument is a RUNTIME value
(not a compile-time literal, so tier 3 can't fold it) — should run **natively** (no WASM tier-5
jump) by memoization. Canonical case: `fib(n)` for runtime `n`, exponential naive → linear memoized.

## What exists (investigated 2026-06-17)

- **`dict<int,int>`** — a scalar-keyed dict, substrate-routed (codegen_pytorch ~808), with no host
  `.item()`. A natural integer-keyed **memo table**.
- **`while_loop` → recurrent neurons** — Sutra's tail-recursive loop with carried `slot` state (the
  `sutra-from-ocaml` `while_sum` fixture lowers `while !i<5 do …` to a `while_loop` + `loop` call;
  state threaded as slots). This is tier 1, native.
- **NO native runtime recursion** — the V1 codegen supports direct calls to VSA builtins +
  tail-recursive loops only; general recursion (a function calling itself) is rejected (it's the
  WASM tier-5 fallback today — the `wasm_core` ran recursive `fib` via its RAM call stack).

So the building blocks for a native memo table (a `dict<int,int>` carried as loop state) + a loop
(recurrent neurons) exist; what's missing is the transformation from tree recursion to a memoizing
loop, since native recursion itself isn't available.

## The tractable realization: tabulation (tree recursion → bottom-up DP loop)

For recursion whose subproblems range over a **regular integer index domain** (fib over `0..n`, and
most "overlapping subproblems" cases), memoization = **tabulation / dynamic programming**: instead
of recursing top-down, compute the subproblems **bottom-up** in a loop, storing each in the memo
table, and return `table[n]`. Crucially, a bottom-up loop **is tail recursion** → recurrent neurons
(tier 1), and the memo table is exactly "the state, growing over time" the spec describes. So:

```
fib(n) = fib(n-1) + fib(n-2)        -- tree recursion, runtime n
   ↓ tabulate
table = dict<int,int>; table[0]=0; table[1]=1
loop i from 2 to n:  table[i] = table[i-1] + table[i-2]      -- a while_loop (recurrent neurons)
return table[n]
```

This stays **native** (no WASM), is linear, and needs no runtime recursion — it reuses tiers 1
(loop) + the existing `dict<int,int>`/array memo. This is the core of tier 4.

## Bounded sub-steps

1. **4a — demonstrate the native target.** Hand-write a tabulated `fib` (a `while_loop` over a
   memo / two carried accumulators) in `.su`, run it on the substrate, confirm `fib(n)` for a
   runtime-style `n` == ground truth — proving the tier-4 native realization is expressible (no
   recursion, no WASM). Grounds everything below in a measured fact. (Surface is the `while_loop` +
   `slot` + `loop`-call form from the `while_sum` fixture, with a variable bound threaded invariant.)
2. **4b — automatic tabulation transform (the real feature).** A compiler pass that DETECTS an
   index-structured tree recursion (`f(n) = combine(f(n-c1), f(n-c2), …)` with a base case and
   structurally-decreasing integer argument) and rewrites it to the bottom-up DP loop over a memo
   table. This is the substantive analysis: recognize the shape, infer the base cases + index step,
   emit the `while_loop`. Verify per shape: `fib`, and one more (e.g. a tribonacci or a binomial)
   run natively == reference.
3. **4c — general-key / irregular memoization (HARDER, may stay partial).** Recursion whose memo
   keys are NOT a regular integer range (tree traversal over arbitrary keys, multi-argument
   recursion) doesn't tabulate into a simple loop. Options: (i) a runtime **worklist** loop that
   carries an explicit agenda + a `dict`/rotation-hashmap memo (still a `while_loop`, but the agenda
   is dynamic); (ii) leave genuinely-irregular recursion on tier 5 (WASM). Decide per-shape; do NOT
   force everything into tabulation. This is where "memoize everything" meets its real limit.

## Open / unclear (flag, don't invent)

- **How far does automatic detection go?** 4b's tabulation is clearly tractable for the
  fib-shaped index-structured case. The GENERAL "memoize everything automatically" claim from the
  chat is not obviously realizable as a single transform — irregular recursion (4c) may need the
  worklist form or WASM. The honest scope: tier 4 makes the **index-structured / overlapping-subproblem
  majority** native via tabulation; the irregular tail stays on tier 5. (Matches the chat's own
  caveat that genuinely tree-shaped non-overlapping recursion "buys you nothing" from memoization.)
- **Memo table as carried loop state vs a persistent device.** `dict<int,int>` carried as `slot`
  state across the `while_loop` is the natural fit (the table IS the recurrent state). Confirm the
  dict survives loop iterations as slot state when 4a is built.

## Why scope before coding

Tier 4 is the hardest tier, with a real analysis (shape detection + tabulation) and a genuinely-
unclear general case. Decomposing 4a (prove the native target) → 4b (the tabulation transform for
the tractable majority) → 4c (the irregular tail, partial) keeps each step bounded and honest about
where automatic memoization stops and tier-5 WASM resumes — rather than over-claiming "memoize
everything" works uniformly.
