# Phase 5.5 tier 3 — fixed-depth multiple recursion → compile-time pre-evaluation: scoping

**Date:** 2026-06-17
**Status:** scoping / design. Tier 3 of the recursion-execution model
(`planning/sutra-spec/recursion-execution-model.md`). Tiers 1–2 are done (tail/loops → recurrent
neurons; single non-tail → tail across all frontends). Tier 3 is a genuinely-new compiler feature
with no existing machinery and an open design problem, so this scopes it before coding (the pattern
that made the JVM/WASM cores go smoothly).

## What exists today (investigated 2026-06-17)

- `simplify.py` — arithmetic **constant folding** (`x+0`→`x`, `x*1`→`x`, subscript-of-literal, etc.)
  + egglog algebraic simplification. Folds *expressions*, does not evaluate *function calls*.
- `loop_desugar.py` — rewrites `loop(cond){body}` into a tail-recursive `iterative_loop` (a substrate
  loop). Produces the loop machinery; does NOT fold a bounded loop to its final value.
- `inliner.py` — inlines single-`return` stdlib functions (arg substitution). Not recursion.

So the spec's "same machinery as loop unrolling (which Sutra is already good at)" is aspirational:
Sutra desugars loops and folds arithmetic, but has **no compile-time evaluation of a bounded
recursive call to a constant**. Tier 3 must add that.

## The detectable class: statically-bounded multiple recursion

A multiple-recursive pure function `f` qualifies for pre-evaluation at a call site `f(args)` when:
1. **`f` is pure** (Sutra functions are — referential transparency makes compile-time evaluation
   safe; this is the whole reason it's allowed).
2. **The recursion is provably bounded** at this call: there is a structurally-decreasing argument
   heading to a base case, and the controlling argument is a **compile-time-known literal** (or
   foldable to one), so the depth is a compile-time constant.
3. **The bound ≤ `max_preeval_depth`** (a compilation argument; see below).

The canonical example is a "no external input" generator like a Fibonacci value `fib(5)` — Emma's
own example of the rare Sutra program whose input is purely self-referential (its starting state).
For such a site the compiler can run `f` at compile time and replace the call with the literal.

## Mechanism: a compile-time partial evaluator (host-side, pure — a legitimate optimization)

Extend the constant-folding layer (`simplify.py`) with a bounded recursive-call evaluator:
- When a call `f(args)` meets the class above, **evaluate `f` symbolically on the AST at compile
  time**, recursing through its body with the literal arguments, up to `max_preeval_depth`. If it
  reaches base cases within the bound, replace the call node with the resulting `IntLiteral` /
  `FloatLiteral`. If it would exceed the bound (or hits a non-foldable construct), **abort the
  pre-eval for that site** and leave the call for the runtime path (tier 4/5).
- This is compile-time, host-side, PURE evaluation of Sutra expressions — the same category as the
  existing arithmetic constant-folding, NOT runtime substrate execution. It does not touch the
  substrate runtime or violate the no-host-readout rule (there is no running program to read from).
- **Memoize the compile-time evaluation** itself (a host dict keyed by `(f, args)`) so that even a
  pre-evaluated `fib(5)` doesn't blow up exponentially at compile time — this mirrors the tier-4
  runtime story and keeps pre-eval cheap.

## The `max_preeval_depth` compilation argument

Per Emma: the maximum pre-evaluation depth is a **compilation argument**, with an
**empirically-tested default**, always present in the project `.toml` (the IDE may auto-fill it).
Above the cap, the site is NOT pre-evaluated (falls through to tier 4 memoization / tier 5 WASM).
The default should be picked by measuring compile-time cost vs runtime savings on representative
programs — a finding to produce when the mechanism exists.

## OPEN DESIGN PROBLEM — *when NOT to pre-evaluate* (Emma flagged this UNSOLVED)

Emma, in the reference chat: deciding when *not* to pre-evaluate at compile time — even when you
technically could — "is actually an unsolved problem in Sutra right now," worth a dedicated
conversation. The tension: pre-evaluation trades **runtime cost → compile-time cost + binary size +
startup time + loss of runtime flexibility**. A program that pre-evaluates a large fixed computation
becomes a giant literal (bloated binary, slow compile) when running it at startup might be cheaper.

This gates the *default policy*, not the mechanism. Candidate policies (for Emma's design pass):
1. **Pre-eval everything up to `max_preeval_depth`** (aggressive; simple; risks binary bloat).
2. **Only pre-eval when the result is "small"** (literal size under a threshold) — bounds bloat.
3. **Require an explicit annotation / pragma** to pre-eval (no automatic; programmer opts in).
4. **Cost-model heuristic:** pre-eval iff (estimated runtime cost saved) > (binary-size + compile-time
   cost added), with the crossover empirically tuned.

→ This is an **Emma-decision** (the severity ladder: surface it, don't guess). It blocks shipping
*automatic* tier-3 pre-eval, NOT the mechanism. Plan: build + verify the mechanism (sub-steps below)
behind an explicit opt-in (policy 3) first; resolve the default policy with Emma before enabling it
automatically. (Captured for the ask-Emma sweep; this is a known-open problem Emma already named,
not a rotting deferred item.)

## Bounded sub-steps (each compile-AND-run, substrate-verified)

1. **3a — detector + compile-time evaluator (opt-in).** A pass that, for a pure recursive call with
   compile-time-literal controlling args and a provable bound ≤ `max_preeval_depth`, evaluates it to
   a literal (memoized at compile time). Gate it behind an explicit flag/annotation (policy 3) so no
   default-policy decision is needed yet. Verify: `fib(5)` (or a fixed-depth tree-recursive fixture)
   folds to its literal AND the folded program runs == the un-folded reference on the substrate
   (the fold must not change observable output — that's the correctness bar).
2. **3b — the `max_preeval_depth` `.toml` argument** + a finding measuring a sane default
   (compile-time cost vs runtime savings on a few fixtures). Above-cap sites fall through cleanly.
3. **3c — resolve the default policy with Emma** (AskUserQuestion: the candidate policies above),
   then enable the chosen default.

## Why scope before coding

Tier 3 adds a compile-time recursive evaluator (new machinery) and rides on an explicitly-unsolved
design problem. Decomposing it (mechanism behind opt-in → measured depth cap → Emma policy decision)
keeps each step bounded and substrate-verifiable, and isolates the one true Emma-decision (the
default policy) from the buildable mechanism — so the build isn't blocked waiting on a conversation.
