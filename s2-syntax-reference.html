S2 Surface Design

Language comparisons before syntax lock-in.

This page is the quick browser view for early S2 language-shape decisions. It compares reference languages, records the current syntactic direction, and keeps live decisions visible while the language is still moving.

Current direction

  • S2 is a substrate-oriented language rather than a class-first application language.
  • Functions exist independently.
  • Declarations use C# signature shape: function vector Add(vector a, vector b) { ... }
  • function. prefix for call-site disambiguation, not declaration.
  • Conditionals use TypeScript-style if (...) { ... } else { ... }.
  • S2 uses return.
  • Looping uses TypeScript-style while (...) { ... }.
  • fuzzy and bool are distinct, and defuzzy(...) converts between them.
  • C# remains a primary readability baseline even where S2 rejects class-first structure.
  • The syntax should stay serious and explicit without borrowing conventional datatype assumptions.
  • Readability matters, but the surface should not pretend the semantics are ordinary imperative programming.

Quick Matrix

The goal is not to imitate one language wholesale. The comparison is here to isolate syntax pressures that matter for S2.

Function declaration pressure

The declaration keyword decision is now made. S2 uses function. The remaining pressure is on signature shape, annotations, and block style.

C#         static Vector Blend(Vector a, Vector b) { ... }
TypeScript function blend(a: Vector, b: Vector): Vector { ... }
Python     def blend(a, b):
Rust       fn blend(a: Vector, b: Vector) -> Vector { ... }
S2         function vector Blend(vector a, vector b) { ... }

S2 calling:
Blend(x, y)              // direct call
function.Blend(x, y)     // disambiguated call

S2 full internal form:
function public static vector Blend(vector a, vector b) { ... }

Most likely current lean

  • Independent functions: yes
  • Declaration shape: function [return] Name([params]) { } (C# signature)
  • Call disambiguation: function.Name()
  • Blocks: braces are the safest current default
  • Conditional form: TypeScript-style if (...) { ... }
  • Loop form: TypeScript-style while (...) { ... }
  • Return keyword: return
  • Truth collapse: defuzzy(...)
  • Truthiness: compilation error on non-bool/fuzzy; geometric via unsafe cast
  • Operator overloading: yes, via function operator
  • Implicit casts: allowed but must be explicitly defined
  • Readability baseline: C#-level explicitness
  • Typing surface: probably lightweight, not conventional datatypes
  • Notation: likely mixed mainstream syntax rather than Lisp-only prefix forms

Established decisions

2026-04-08: Functions exist independently

S2 treats functions as top-level constructs. Future grouping features, if added, should organize functions rather than own them. Method syntax should be secondary sugar, not the core mental model.

2026-04-08: Use function for declarations

S2 aligns with TypeScript here. The keyword is explicit, readable, and compatible with the independent-function model without pulling in class-first structure.

2026-04-08: Use TypeScript-style conditionals and loops

S2 uses if (...) { ... } else { ... } and while (...) { ... } as its current baseline control-flow forms.

2026-04-08: Use return

S2 uses return for explicit return points and early exits.

2026-04-08: Distinguish fuzzy and bool

defuzzy(...) converts a fuzzy into a bool, even though both are vector-level substrate values underneath.

2026-04-08: if (cat) is a compilation error

Classes do not exist at runtime. Branching requires bool or fuzzy. To get truthiness from an arbitrary vector, use unsafeCast<fuzzy>(x) explicitly.

2026-04-08: Operators support overloading

Declared as function operator +(vector a, vector b) { ... }. Vector operations like bind and bundle map naturally to arithmetic operators.

2026-04-08: Truthiness is geometric, not type-level

Truthiness is euclidean distance from true/false vectors in embedding space. It is a runtime property accessed only via unsafe cast or unsafeOverride, never automatic.

2026-04-08: Implicit casts are allowed but must be explicitly defined

Implicit conversions can happen at call sites, but the conversion itself must be declared explicitly in the source. No silent coercions without a visible definition.

Questions still open

  • What should the rest of the function signature look like?
  • Should files imply namespaces, or should grouping be explicit?
  • How should primitive substrate operations read in source?
  • Does S2 need any annotation layer for semantic roles?
  • Expression-versus-statement bias?
  • Return annotation syntax?

TypeScript's Competing Styles (why C# is the stronger readability baseline)

TypeScript inherits JavaScript's multiple ways to write equivalent code. This is useful context for understanding why S2 leans on C# for readability discipline while borrowing TypeScript's tooling model.

── Function declaration ──────────────────────────────────────────────

Style 1: function declaration (S2 chose this)
function blend(a: Vector, b: Vector): Vector {
    return combine(a, b);
}

Style 2: arrow function assigned to const
const blend = (a: Vector, b: Vector): Vector => {
    return combine(a, b);
};

Style 3: arrow function with implicit return
const blend = (a: Vector, b: Vector): Vector => combine(a, b);

Style 4: method in an object literal
const ops = {
    blend(a: Vector, b: Vector): Vector {
        return combine(a, b);
    }
};

Style 5: method in a class
class Ops {
    blend(a: Vector, b: Vector): Vector {
        return combine(a, b);
    }
}

── C# comparison: ONE dominant form ──────────────────────────────────

static Vector Blend(Vector a, Vector b)
{
    return Combine(a, b);
}
// That's it. There is one way to write this.

── Why this matters for S2 ───────────────────────────────────────────

TypeScript's five styles all do the same thing, but they look
different, scan differently, and create ecosystem variance in
what "normal" code looks like.

C# has one dominant form per construct. This is why C# is the
readability baseline even where S2 borrows TypeScript's
function keyword and tooling model.

S2 should have ONE way to declare a function: function name(...) { }

Open Syntactic Choices

Each card below is a decision that needs to be made. Examples show how reference languages handle it and what S2 candidates look like.

Anonymous functions / lambdas

Current lean: lambda keyword. Exact form undecided.

C#
Func<Vector, Vector> op = (a) => Transform(a);
items.Select(x => Transform(x));

TypeScript
const op = (a: Vector): Vector => transform(a);
items.map(x => transform(x));

Python
op = lambda a: transform(a)
list(map(lambda x: transform(x), items))

Rust
let op = |a: Vector| -> Vector { transform(a) };
items.iter().map(|x| transform(x));

S2 candidates
var op = lambda(vector a) { return Transform(a); };
var op = lambda vector(vector a) { return Transform(a); };
var op = (vector a) => Transform(a);
Map(items, lambda(vector x) { return Transform(x); });

Key question: does lambda feel right as a keyword, or is it too Python? Should lambdas have the same return-type-before-name shape as functions?

Variable declaration keyword

How do you bind a name to a value?

C#
var result = Blend(a, b);
Vector result = Blend(a, b);

TypeScript
const result = blend(a, b);
let result = blend(a, b);

Python
result = blend(a, b)

Rust
let result = blend(a, b);
let mut result = blend(a, b);

S2 candidates
var result = Blend(a, b);          // C#-style, inferred type
vector result = Blend(a, b);       // explicit type, no keyword
let result = Blend(a, b);          // Rust-style
result = Blend(a, b);              // bare assignment (Python-style)

C# uses var for inferred and explicit type for declared. S2 already shows var in some examples. Is var the direction? Is mutability tracked?

Mutability

Are bindings mutable by default or immutable by default?

C#         var x = 5; x = 6;          // mutable by default
TypeScript let x = 5; x = 6;          // let = mutable
           const x = 5; x = 6;        // const = immutable (error)
Rust       let x = 5; x = 6;          // immutable by default (error)
           let mut x = 5; x = 6;      // explicit mutable

S2 candidates
var x = Blend(a, b); x = Step(x);          // mutable by default (C#)
let x = Blend(a, b); x = Step(x);          // immutable by default (Rust-style, error)
let mut x = Blend(a, b); x = Step(x);      // explicit mutable
val x = Blend(a, b);                       // immutable (Kotlin/Scala)
var x = Blend(a, b);                       // mutable

Vector operations often chain transformations. Does S2 favor immutable pipelines or mutable state updates?

Namespace / module / file organization

How are functions grouped? Does a file imply a namespace?

C#
namespace S2.Core;
public static class Basis {
    public static Vector Blend(Vector a, Vector b) { ... }
}
// Call: Basis.Blend(a, b)

TypeScript
export function blend(a: Vector, b: Vector): Vector { ... }
// Call: import { blend } from './core';

Rust
pub fn blend(a: Vector, b: Vector) -> Vector { ... }
// Call: core::blend(a, b)

Python
def blend(a, b): ...
# Call: from core import blend

S2 candidates
// Option 1: file IS the namespace
// file: core.s2
function vector Blend(vector a, vector b) { ... }
// Call: core.Blend(a, b) or just Blend(a, b) if imported

// Option 2: explicit namespace declaration
namespace Core;
function vector Blend(vector a, vector b) { ... }
// Call: Core.Blend(a, b)

// Option 3: no namespaces, function. prefix only
function vector Blend(vector a, vector b) { ... }
// Call: Blend(a, b) or function.Blend(a, b)

This is load-bearing for how S2 projects feel. C# has solutions/projects/namespaces. Does S2 need that structure?

Visibility / access modifiers

The full internal form has public static. What shows in surface syntax?

C#
public static Vector Blend(Vector a, Vector b) { ... }
private static Vector Helper(Vector v) { ... }
internal static Vector Util(Vector v) { ... }

Rust
pub fn blend(a: Vector, b: Vector) -> Vector { ... }
fn helper(v: Vector) -> Vector { ... }  // private by default

TypeScript
export function blend(a: Vector, b: Vector): Vector { ... }
function helper(v: Vector): Vector { ... }  // module-private

S2 candidates
// Option 1: everything public by default, mark private
function vector Blend(vector a, vector b) { ... }
private function vector Helper(vector v) { ... }

// Option 2: private by default, mark public (Rust-like)
function vector Helper(vector v) { ... }
public function vector Blend(vector a, vector b) { ... }

// Option 3: export keyword (TypeScript-like)
export function vector Blend(vector a, vector b) { ... }
function vector Helper(vector v) { ... }

Which default feels right for a substrate language? C# is explicit everywhere. Rust defaults to private. TypeScript uses export.

Method-style calls on values

Can you call functions with dot syntax on values?

C#
var result = signal.Transform();
var result = Transform(signal);

Rust
let result = signal.transform();  // method via impl
let result = transform(signal);   // free function

TypeScript
const result = signal.transform();
const result = transform(signal);

S2 candidates
// Option 1: no dot calls, free functions only
var result = Transform(signal);

// Option 2: dot calls as sugar for first-arg dispatch
var result = signal.Transform();
// desugars to: Transform(signal)

// Option 3: dot calls only on primitives
var result = signal.defuzzy();
// but: var result = Blend(a, b);  // no dot form

Dot syntax is ergonomic for chaining. But it implies objects, which S2 doesn't have at runtime. Is dot sugar acceptable?

For loops / iteration

while is decided. Does for exist?

C#
foreach (var item in collection) { ... }
for (int i = 0; i < n; i++) { ... }

TypeScript
for (const item of collection) { ... }
for (let i = 0; i < n; i++) { ... }

Python
for item in collection: ...
for i in range(n): ...

Rust
for item in collection { ... }
for i in 0..n { ... }

S2 candidates
for (var item in collection) { ... }
for (var i = 0; i < n; i++) { ... }
// or: no for loops, only while + iterators
// or: for-each only, no C-style for

Does iterating over collections make sense when the substrate is vectors? What would a collection even be — a matrix? A tuple?

Error handling

What happens when something fails?

C#         try/catch/finally
TypeScript try/catch/finally
Python     try/except/finally
Rust       Result<T, E> + ? operator

S2 candidates
// Option 1: try/catch (C#-style)
try {
    var v = UnsafeCast<cat>(animal);
} catch {
    return fallback;
}

// Option 2: Result type (Rust-style)
var v = UnsafeCast<cat>(animal)?;

// Option 3: fuzzy failure
// failures return a vector near "false"
// no explicit error handling needed

Option 3 is the most S2-native: if everything is vectors, failure is just distance from expected results. But unsafe ops might need hard errors.

Comments

What do comments look like?

C#         // line comment    /* block */
TypeScript // line comment    /* block */
Python     # line comment     """ docstring """
Rust       // line comment    /* block */   /// doc

S2 candidates
// Option 1: C-family (matches braces decision)
// line comment
/* block comment */

// Option 2: include doc comments
/// Doc comment for function below
function vector Blend(vector a, vector b) { ... }

C-family comments are the obvious choice given brace syntax. Do doc comments exist?

Generics / parameterized types

Does S2 have generics? The class system is user-defined, so this matters.

C#         List<Vector>    T Blend<T>(T a, T b)
TypeScript Array<Vector>   function blend<T>(a: T, b: T): T
Rust       Vec<Vector>     fn blend<T>(a: T, b: T) -> T

S2 candidates
// Option 1: angle brackets (C#-style)
function T Blend<T>(T a, T b) { ... }

// Option 2: no generics — everything is vectors
// if everything reduces to vectors at runtime,
// generics may be a compile-time-only concern

// Option 3: constrained generics
function T Blend<T: vector>(T a, T b) { ... }

Since classes are compile-time and vectors are runtime, generics might only matter for the compiler's acceptance checking. unsafeCast already uses angle brackets.

Chaining / piping

How do you compose multiple transformations?

C#
var result = Step3(Step2(Step1(input)));
// or with LINQ
var result = input.Step1().Step2().Step3();

TypeScript
const result = step3(step2(step1(input)));
// or with pipe proposal
const result = input |> step1 |> step2 |> step3;

Rust
let result = input.step1().step2().step3();

F#
let result = input |> step1 |> step2 |> step3

S2 candidates
// Option 1: nested calls
var result = Step3(Step2(Step1(input)));

// Option 2: pipe operator
var result = input |> Step1 |> Step2 |> Step3;

// Option 3: dot chaining (if method sugar exists)
var result = input.Step1().Step2().Step3();

Vector transformations naturally chain. A pipe operator would fit the substrate model well without requiring dot syntax.
